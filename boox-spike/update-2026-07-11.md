# Update 2026-07-11 — V7 繪圖配方落地：量測格線＋讀題三步驟＋稀疏錨點筆法

把 2026-07-10 在 stroke-llm 實驗場（`stroke-llm/experiments/{digit-prompt,math,people}/`）
用 gpt-5.5（reasoning effort low）驗證出的「V7 配方」移植進 boox-spike 的 oracle 管線。
使用者側零改變：頁面上仍然只寫白話（「幫我把答案寫上去」「畫個朋友跟他牽手」），
所有工程都在系統 prompt 與快照前處理裡。

## 實驗依據（為什麼要改）

| 發現 | 數據 |
| --- | --- |
| 視覺模型在空白紙上讀絕對座標會系統性漂移 | 讀答案線位置偏 ~150px（frame 的 ~7%） |
| 快照印上格線刻度後，同一讀取誤差大幅縮小 | ~150px → ~45px（約 3 倍改善） |
| 稀疏地標錨點＋過點 spline 是四種編碼中字形最佳、最穩 | 對照密集點/DELTA/SketchAgent 全勝（見 all-encodings-sheet.png） |
| 尖點（cusp）錨點不加倍會被 spline 抹圓 | 數字 3 退化成 S 形；加倍後兩輪全成形 |
| 白話請求下，low effort 會草率讀題 | 3×5 誤讀率 2/2 → 加「逐字＋筆畫證據」轉錄規則後 1/3 |
| 「把需求翻譯成自己的規則」可由模型自己做 | 「牽手」自行推導出「終點距 (575,490) ≤10px」並執行成功 |

## 改動內容

### 1. `TomView.kt` — 快照疊量測格線（前處理）

- 新增 `drawMeasuringGrid()`：淡灰線每 **50 snapshot px** 一條、每 100px 標座標數字
  （上下緣標 x、左右緣標 y），畫在**墨水底下**，且只進 `capturePagePng()` 的快照 —
  **頁面本身永遠不會出現格線**。
- `capturePagePng()` 在 `drawColor(WHITE)` 之後、疊墨水圖層之前呼叫它。
- 順手修正該函式過期的 doc comment（原文還在講已廢棄的「100×100 reply grid」協議）。

### 2. `Oracle.kt` `SYSTEM_PROMPT` — 三處升級（文法 token 不動，parser 零改動）

- **格線說明**（新段落）：告知模型快照上有量測格線、只是量尺不是頁面內容、
  讀任何位置前先對照標籤、絕不提及也絕不描畫。
- **SEE 規則改為三步驟**（V7 的 READ→ANSWER→PLAN，全部住進現成的 SEE 區塊）：
  1. **READ**：逐字轉錄，潦草字先描述筆畫形狀當證據再定字（「兩個朝左開口的疊碗 → 3」），
     模稜兩可的字要並列候選、憑筆畫證據挑；
  2. **ANSWER**：頁面內容若在問問題（算式、填空），先在 SEE 內給出答案並複核一次算術；
  3. **PLAN**：動筆前在 SEE 裡寫繪圖計畫 — 要依附的既有墨水（從格線讀出的像素位置）、
     外框、自訂約束（接觸點 ~10px、包含關係）、筆徑；之後的筆畫必須遵守自己的計畫。
- **Stroke craft 從「密集錨點」改為「稀疏地標錨點」**（配合 `ReplyDsl.densify` 的
  Catmull-Rom 過點 spline）：
  - 錨點放地標（起點、極值、轉角、終點）；每個彎至少 4 錨；直線只要兩端；
  - **尖點/反轉處錨點連寫兩次**（不加倍會被 spline 抹圓 — 實驗中數字 3 的死因）；
  - 新增 worked example：字母 B（直脊＋兩個右凸碗＋中央加倍尖點）—
    實驗顯示「結構同、形狀異」的示範能帶起字形品質且不會形狀滲透。
- `turnPrompt()` 每回合補一句「Read positions against the printed gray grid labels.」。

## 預期效果

- Tom 畫東西、寫字的落點誤差從 ~7% frame 降到 ~2%；
- 手寫字形（數字、字母）成形穩定，尖點不再糊掉；
- 「在某某旁邊畫」「跟他牽手」這類關係型請求可靠（接觸點自推導）；
- 讀潦草手寫的誤讀率下降（但未歸零 — low effort 的殘餘誤讀約 1/3，
  要再壓可考慮升 reasoning effort 或分離轉錄呼叫）。

## 驗證方式

1. `./gradlew test` — JVM 單元測試（OracleTest / ReplyDslTest / ScriptTest）。
2. 裝置驗證：寫「3×5=＿」等 2.8 秒 → 拉 `files/last-page.png` 確認格線在快照上
   且頁面本身乾淨；看 logcat `riddle-spike` 的 SEE 是否出現逐字轉錄與 PLAN；
   答案應寫在線上、字形端正。
3. 關係型驗證：畫一個無臉火柴人，寫「幫他畫笑臉，再畫一個牽他手的朋友」。

## 回滾

三處改動彼此獨立：拿掉 `drawMeasuringGrid` 呼叫即回到無格線快照；
`SYSTEM_PROMPT` 各段可單獨還原。文法 token 與 parser 完全未動。

# 追加（同日）— Drawing-only 模式：回覆只有圖，沒有文字

需求：LLM 不再用 TEXT 區塊回話，一切回應都是筆畫；要「說」的內容（答案、標籤）
用筆畫逐字寫出（稀疏錨點筆法本來就會寫字）。

## 改動

- `Oracle.kt` `SYSTEM_PROMPT`：文法區塊移除 `TEXT x y … END_TEXT`；刪除 TEXT 規則；
  「可以只回話不畫」改為「**筆墨是唯一的聲音**：每回合至少畫一筆；需要文字就用
  筆畫親手寫」。`turnPrompt()` 不再報 TEXT 行高。SEE（私有筆記/轉錄/計畫）保留 —
  它不上頁面，是模型的記憶與規劃空間。
- `TomView.kt`：`onOracleBlock()` 靜默丟棄不聽話的 TEXT 區塊（logcat 仍記錄）；
  `writeExcuse()` 改走**狀態列**而非把致歉句寫上頁面，並恢復 IDLE／解除 rawPen
  暫停，避免失敗回合卡住頁面。
- `OracleTest.kt`：turnPrompt 斷言改為「不再提及 TEXT 行高」。
- Parser（`ReplyDsl`）刻意不動：TEXT 解析保留當容錯層，protocol 由 prompt 收窄。

## 回滾

還原 SYSTEM_PROMPT 的三處字串與 TomView 的兩處小改即可；資料層無變化。
